# Pipewise 项目功能分析与优化建议

## 1. 项目定位

`Pipewise` 是一个围绕 `pandas.DataFrame` 构建的轻量数据处理管道类。它的核心目标不是替代 `pandas`，而是把“按列声明输入、按步骤串联处理、自动回写结果列”的逻辑封装成统一的注册式流水线，减少业务代码里重复的列选择、逐行处理和中间结果管理。

当前项目代码集中在 [core.py](/Users/xiaoyuyu/Documents/work/dev/datapipe/pipewise/core.py)，已经具备一个可直接使用的最小可用版本。

## 2. 主要功能

按业务价值和代码主线来看，项目当前主要能力如下：

### 2.1 任务注册与串行执行

- 通过 `@pipewise.register(...)` 装饰器注册任务。
- 函数参数名会自动映射到 DataFrame 同名列。
- 多个任务按照注册顺序执行，后续任务可以直接依赖前序任务新增的列。

这部分是整个项目的核心能力，决定了它是否能作为“声明式数据处理流水线”使用。

### 2.2 多种输出模式

`register(outputs=...)` 目前支持以下几种返回写回模式：

- `None`：仅执行副作用逻辑，不写回 DataFrame。
- `"col"`：单列输出。
- `["col1", "col2"]`：固定多列输出。
- `{"col": type}`：带类型声明的输出，执行后自动 `astype`。
- `"dict"`：动态字典输出，允许任务按条件生成不同列。

这部分让 `Pipewise` 能覆盖从简单派生列到复杂规则映射的大多数处理场景。

### 2.3 向量化优先，失败后自动回退到逐行执行

- 默认 `vectorized=True`，优先把整列 `Series` 传给函数。
- 若函数中包含不能直接处理 `Series` 的分支逻辑，例如 `if a < 10`，会在异常后自动回退为 `row-wise` 模式。
- 也支持显式指定 `vectorized=False` 强制逐行执行。

这是当前实现里最有特色的能力，兼顾了性能和易用性。

### 2.4 GroupBy 分组执行

- 支持 `groupby="col"` 或 `groupby=["c1", "c2"]`。
- 每个分组独立执行同一处理函数，再把结果合并回原表。

这使 `Pipewise` 不只是一个普通列变换器，也能处理分组规则、组内计算和条件判断。

### 2.5 任务管理与可观测性

- `tasks`：查看已注册任务摘要。
- `remove(func)`：删除单个任务。
- `clear()`：清空任务。
- `plan()`：打印执行计划。

这部分能力对调试和交互式开发比较友好，适合 notebook 或快速脚本使用。

### 2.6 失败回滚

- `run()` 执行前会保存快照。
- 任一任务失败时，会把本次执行期间的修改全部回滚。
- 对外统一抛出 `RuntimeError`，并保留原始异常上下文。

这让 `Pipewise` 在多步骤处理场景里更安全，避免中间状态污染原始数据。

## 3. 代码结构介绍

当前代码虽然集中在一个文件内，但职责划分相对清晰：

### 3.1 对外接口

- `Pipewise.__init__(data)`：绑定源 DataFrame。
- `Pipewise.register(...)`：注册任务。
- `Pipewise.run(inplace=False)`：执行流水线。
- `Pipewise.tasks / remove / clear / plan`：任务管理与查看。

### 3.2 内部辅助逻辑

- `_normalize_outputs()`：解析输出配置。
- `_rollback()`：执行失败后的数据恢复。
- `_parse_inputs()`：从函数签名提取输入列和 `**kwargs` 模式。

### 3.3 执行分发逻辑

- `_execute()`：总调度入口。
- `_execute_vectorized()`：向量化执行。
- `_execute_rowwise()`：逐行执行。
- `_execute_groupby()`：分组执行。
- `_apply_types()`：输出列类型转换。

### 3.4 当前设计特点

- 设计上偏“单类集中式实现”，上手简单。
- 通过少量内部 helper 完成模式切换，整体可读性尚可。
- 对小项目和单文件复用很友好，但随着功能继续增长，单文件会逐步变得难维护。

## 5. 按优先级排序的优化建议

下面的建议按“优先级 + 对实际使用影响”排序。

### P0：修正异常回退策略，避免吞掉真实业务错误

当前行为：

- 只要 `vectorized=True` 的任务报错，就会直接回退到逐行执行。
- 这会把“函数不支持 `Series`”和“函数本身逻辑真的出错了”混在一起。

风险：

- 如果向量化函数内部有真实 bug，例如除零、KeyError、自定义逻辑错误，框架会悄悄回退到逐行执行。
- 结果可能不是立即失败，而是性能下降、行为改变，增加排查难度。

建议：

- 只对明确的“向量化不兼容”异常做回退，例如 `ValueError: The truth value of a Series is ambiguous` 这类典型错误。
- 为 `register()` 增加参数，例如 `fallback_on_vectorized_error=True/False`。
- 在回退时记录 warning 或 debug 信息，告诉调用方发生过执行模式切换。

这是最优先的改进点，因为它直接影响结果可信度和线上排障。

### P1：补充更严格的输出校验

当前行为：

- 多列输出时会检查返回值数量。
- 但动态字典模式、向量化模式下的返回类型校验仍然偏宽松。

建议：

- 对 `"dict"` 模式统一校验返回值必须是 `dict`、`DataFrame` 或可展开对象。
- 对多输出向量化模式显式校验长度、列数和索引是否匹配。
- 对类型声明模式增加更清晰的错误提示，包含原始值样本和目标类型。

这会明显提升接口稳定性，降低“静默写错列”的风险。

### P1：把任务元信息结构化，替代裸 `dict`

当前行为：

- `_tasks` 中每个任务都用普通字典保存。

建议：

- 使用 `dataclass` 或轻量任务对象封装任务元信息。
- 给字段增加明确类型，如 `outputs`、`groupby`、`vectorized`、`dict_mode`。

好处：

- 降低魔法字符串使用频率。
- 提升 IDE 提示和维护性。
- 为后续增加 `name`、`description`、`enabled`、`retry_policy` 等元数据留空间。

### P2：分离模块职责，拆分为多文件结构

当前行为：

- 所有逻辑都在单文件中。

建议：

- `core.py`：公开 `Pipewise` 类。
- `executors.py`：向量化、逐行、groupby 执行器。
- `models.py`：任务定义。
- `errors.py`：自定义异常。
- `tests/`：测试代码目录。

这不是最急的，但对后续扩展非常重要，尤其是当项目要支持更多执行策略时。

### P2：增强可观测性与调试能力

建议：

- 为 `run()` 增加 `verbose` 或日志回调。
- 返回执行报告，例如每个任务的执行模式、耗时、生成列、是否发生回退。
- `plan()` 可以支持返回字符串，而不只是直接 `print()`。

这会让项目更适合接入正式数据处理流程，而不仅是手工脚本。

### P3：补充性能与边界测试

当前测试已覆盖主流程，但仍建议增加：

- 大数据量下向量化与逐行模式的性能对比测试。
- 空 DataFrame、重复列名、缺失值、混合类型、MultiIndex 等边界测试。
- GroupBy 后索引顺序与结果对齐测试。

这部分优先级较低，但对库化发布很有帮助。

## 6. 结论

当前 `Pipewise` 已经具备清晰的核心价值：

- 把 DataFrame 处理逻辑组织成可注册、可串联的任务流水线。
- 在易用性和性能之间做了不错的平衡。
- 已支持多输出、分组执行、类型转换、任务管理和失败回滚。

如果下一步要继续演进，我建议优先处理两件事：

1. 收紧“向量化失败自动回退”的判定条件。
2. 强化输出校验和任务结构建模。

这两项完成后，项目会从“好用的单文件工具”更稳地迈向“可维护的小型数据处理库”。
