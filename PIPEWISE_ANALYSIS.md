# Pipewise 项目功能

## 1. 项目定位

`Pipewise` 是一个围绕 `pandas.DataFrame` 构建的轻量数据处理管道类。它的核心目标不是替代 `pandas`，而是把”按列声明输入、按步骤串联处理、自动回写结果列”的逻辑封装成统一的注册式流水线，减少业务代码里重复的列选择、逐行处理和中间结果管理。

## 2. 主要功能

按业务价值和代码主线来看，项目当前主要能力如下：

### 2.1 任务注册与串行执行

- 通过 `@pipewise.register(...)` 装饰器注册任务。
- 函数参数名会自动映射到 DataFrame 同名列。
- 多个任务按照注册顺序执行，后续任务可以直接依赖前序任务新增的列。

这部分是整个项目的核心能力，决定了它是否能作为“声明式数据处理流水线”使用。

### 2.2 多种输出模式

`register(outputs=...)` 目前支持以下几种返回写回模式：

- `None`：仅执行副作用逻辑，不写回 DataFrame。
- `"col"`：单列输出。
- `["col1", "col2"]`：固定多列输出。
- `{"col": type}`：带类型声明的输出，执行后自动 `astype`。
- `"dict"`：动态字典输出，允许任务按条件生成不同列。

这部分让 `Pipewise` 能覆盖从简单派生列到复杂规则映射的大多数处理场景。

### 2.3 向量化优先，失败后自动回退到逐行执行

- 默认 `vectorized=True`，优先把整列 `Series` 传给函数。
- 若函数中包含不能直接处理 `Series` 的分支逻辑，例如 `if a < 10`，会在异常后自动回退为 `row-wise` 模式。
- 也支持显式指定 `vectorized=False` 强制逐行执行。

这是当前实现里最有特色的能力，兼顾了性能和易用性。

### 2.4 GroupBy 分组执行

- 支持 `groupby="col"` 或 `groupby=["c1", "c2"]`。
- 每个分组独立执行同一处理函数，再把结果合并回原表。

这使 `Pipewise` 不只是一个普通列变换器，也能处理分组规则、组内计算和条件判断。

### 2.5 任务管理与可观测性

- `tasks`：查看已注册任务摘要。
- `remove(func)`：删除单个任务。
- `clear()`：清空任务。
- `plan()`：打印执行计划。

这部分能力对调试和交互式开发比较友好，适合 notebook 或快速脚本使用。

### 2.6 失败回滚

- `run()` 执行前会保存快照。
- 任一任务失败时，会把本次执行期间的修改全部回滚。
- 对外统一抛出 `RuntimeError`，并保留原始异常上下文。

这让 `Pipewise` 在多步骤处理场景里更安全，避免中间状态污染原始数据。

## 3. 代码结构介绍

当前代码虽然集中在一个文件内，但职责划分相对清晰：

### 3.1 对外接口

- `Pipewise.__init__(data)`：绑定源 DataFrame。
- `Pipewise.register(...)`：注册任务。
- `Pipewise.run(inplace=False)`：执行流水线。
- `Pipewise.tasks / remove / clear / plan`：任务管理与查看。

### 3.2 内部辅助逻辑

- `_normalize_outputs()`：解析输出配置。
- `_rollback()`：执行失败后的数据恢复。
- `_parse_inputs()`：从函数签名提取输入列和 `**kwargs` 模式。

### 3.3 执行分发逻辑

- `_execute()`：总调度入口。
- `_execute_vectorized()`：向量化执行。
- `_execute_rowwise()`：逐行执行。
- `_execute_groupby()`：分组执行。
- `_apply_types()`：输出列类型转换。

### 3.4 当前设计特点

- 设计上偏“单类集中式实现”，上手简单。
- 通过少量内部 helper 完成模式切换，整体可读性尚可。
- 对小项目和单文件复用很友好，但随着功能继续增长，单文件会逐步变得难维护。

## 4. 测试案例说明

测试覆盖的主要场景如下：

1. 向量化多列输出：验证固定多输出列的基础行为。
2. 自动回退：验证分支型函数从向量化回退到逐行执行。
3. 动态字典输出：验证不同分支按需生成列。
4. GroupBy：验证分组后的 split-apply-combine 流程。
5. 类型声明输出：验证自动类型转换。
6. `inplace` 行为：验证默认不改原表，以及原地更新模式。
7. 任务管理：验证 `tasks`、`remove`、`clear`。
8. 执行计划打印：验证 `plan()` 的输出内容。
9. 回滚机制：验证任务失败后数据恢复。
10. `**kwargs` 逐行模式：验证额外列透传。
11. 混合模式：验证向量化任务和逐行任务串联执行。

## 5. 结论

当前 `Pipewise` 已经具备清晰的核心价值：
- 把 DataFrame 处理逻辑组织成可注册、可串联的任务流水线。
- 在易用性和性能之间做了不错的平衡。
- 已支持多输出、分组执行、类型转换、任务管理和失败回滚。
